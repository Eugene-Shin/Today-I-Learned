## 2차시(1/7) - GitHub Flow
### GitHub Flow 전략
- master branch
- feature branch - 작은 단위의 기능 개발 후 push해 pr 받은 후 병합되면 브랜치 삭제

### GitHub Fork할 때
1. fork 후 clone
2. git remote add upstream으로 상위 레포와 연결
3. git remote -v로 확인하면![[스크린샷 2026-01-07 오전 10.48.36.png]]
4. upstream에 추가된 내용 있으면 현재 내가 수정한 내용 git stash로 가리고 pull해 온 다음 git stash pop해서 충돌나는 부분 resolve
### Git 명령어 사용
- git checkout -b 쓰지 말고 git branch와 git switch 쓰기
- 보고 있는 브랜치 변경시 충돌날 때도 stash로 가리고 이동 후 stash pop하고 git resolve하면 됨
	- 충돌 발생할 경우에는 해결 후에 git stash drop을 수행해야만 함
- .env(api key) 같은 거 공유할 때
	1. private repository 만들고 submodule로 지정
	2. cd로 해당 폴더로 이동해서 수정하고
		1. submodule에서 한 번 commit, push
		2. 바깥 레포지토리에서도 commit, push해야 submodule의 바뀐 해시값을 바라보게 됨
		3. 이후 pr
- 히스토리 관리 명령어
	- git reset
		- git reset --soft HEAD~1 : 마지막 커밋만 취소, 수정 내용은 staging에 그대로
		- git reset --mixed HEAD~1 : 마지막 커밋 취소하고 수정 내용도 unstaged로
		- git reset --hard HEAD~1 : 마지막 커밋 완전 삭제
	- git reflog
		- HEAD 로그 전부 기록
		- reset / rebase / checkout 등으로 사라진 커밋도 추적 가능
		- a3f1c2d HEAD@{0}: reset: moving to HEAD~1
		  b9d8e7f HEAD@{1}: commit: add login logic
		  위와 같은 결과 나왔을 때
		  git reset --hard HEAD@{1}로 복구
	- git rebase <기준 브랜치>
		- 현재 브랜치의 커밋들을 기준 브랜치의 최신 커밋 위로 옮김
		- branch의 commit들을 재배치해서 히스토리를 일직선으로 깔끔하게 만듦
- git push --prune하면 내 로컬 branch들을 날려줌(가장 많이 쓰는 옵션)

--- COMMIT TYPE ---
  <타입> 리스트
  :sparkles: Feat           : 기능 (새로운 기능)
  :pencil2: Fix             : 버그 (버그 수정)
  :art: Refactor            : 리팩토링
  :ambulance: !Hotfix       : 핫픽스
  :memo: Docs               : 문서 수정 (문서 추가, 수정, 삭제, README)
  :wrench: Chore            : 기타 변경사항 (빌드 스크립트 수정, assets, 패키지 매니저 등)
  :label: Rename            : 파일 혹은 폴더명을 수정하거나 옮기는 작업만 한 경우
  :coffin: Remove           : 파일을 삭제하는 작업만 수행한 경우
  :rewind: Revert           : 변경 사항 되돌리기
  :lipstick: Style          : 스타일 변경 (코드 포맷팅, 세미콜론 누락 등 기능에 영향을 주지 않는 변경)
  :construction_worker: CI  : CI 설정 및 스크립트 수정
 ------------------
  <제목 - Subject Rule>
  명령문으로
  특수문자 금지
  제목과 본문을 한 줄 띄워 분리하기
  개조식 구문으로 작성(간결하고 요점적인 서술)
------------------
  <본문 - Body Rule>
  본문은 "어떻게" 보다 "무엇을", "왜"를 설명
  최대한 자세히 작성
  한줄당 72자 내로 작성
  본문에 여러줄의 메시지를 작성할 땐 "-"로 구분
  쓸 필요가 없으면 생략 가능
 ------------------
  <꼬리말 - Footer Rule>
  필수가 아닌 optioanl
  유형: #이슈번호의 형식으로 작성
  이슈 트래커 ID 작성
  여러개의 이슈 번호는 ,로 구분
  ex) Fixes: #47 Related to: #32, #21
--- ISSUE TRACKER ---
  Resolves     : 이슈 해결했을 때 사용
  Ref          : 참고할 이슈가 있을 때 사용
  Related to   : 해당 커밋에 관련된 이슈번호 (아직 해결되지 않은 경우)
--- How To Use ---
  git config --global commit.template ./.gitmessage.txt
